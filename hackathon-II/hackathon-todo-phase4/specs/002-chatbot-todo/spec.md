## Clarifications

### Session 2025-02-12

- **Q1: Feature scope boundaries** → **A: Strict MVP** - Feature scope is limited to core task CRUD operations only. Explicitly OUT of scope for Phase 3: recurring tasks, task priorities, due dates, tags, attachments, task sharing/collaboration, external calendar integrations, and advanced task attributes. Only the 5 MCP tools (add_task, list_tasks, complete_task, delete_task, update_task) and 8 natural language patterns focused on basic operations are included.
- **Q2: Concurrent user access** → **A: PostgreSQL ACID transactions** - Concurrency is handled at the database level through ACID transaction isolation provided by Neon Serverless PostgreSQL. Each chat request is stateless and independent; race conditions are resolved by the database's built-in concurrency control. No application-level locking, version fields, or optimistic concurrency control is required.
- **Q3: AI service unavailability** → **A: HTTP 503 with user-friendly message** - When OpenAI service fails or times out, the chat endpoint returns HTTP 503 status code with message: "AI assistant temporarily unavailable. Please try again." The user's message is already persisted to conversation history before agent invocation. Error is logged to console without stack traces. No retry logic or circuit breaker implemented; user can retry their message manually.
- **Q4: Message‑Task relationship** → **A: No direct linking (use tool_calls)** - Avoid adding `task_id` foreign key to the Message model to preserve its primary role as a conversation log. Instead, leverage the existing `tool_calls` response field, which already contains `id`, `title`, `status`, and other operation metadata for every task action. If future audit needs arise, parsing the assistant's response text or extending with a separate `OperationLog` table in Phase 4/5 can provide richer lineage tracking without altering core Message semantics in Phase 3.
- **Q5: Data retention policy** → **A: Unlimited retention** - All conversations and messages are persisted indefinitely in Neon DB. No automatic cleanup, archival, or deletion mechanisms are implemented in Phase 3. Users cannot delete individual messages or conversation history. This aligns with the requirement that "conversations persist after server restarts" and supports full audit trail capabilities. Database models include `created_at` and `updated_at` timestamps but no `deleted_at` or `archived_at` fields.
